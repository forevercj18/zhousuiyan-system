# 订单 × 转寄任务 × 单套库存 三级联动总图（V1）

日期：2026-03-11  
用途：统一说明订单、转寄任务、单套库存三层对象的职责边界、驱动关系、典型闭环和异常口径。  
适用范围：运营培训、需求评审、开发落地、测试验收。

---

## 1. 为什么需要这份总图

当前系统已经不是单纯“订单管理”系统，而是一个包含：

1. 订单履约流转
2. 转寄任务调度
3. 单套库存链路追踪

三层联动的业务系统。

如果只看其中一层，会产生典型误解：

1. 只看订单，不知道库存到底去哪了
2. 只看转寄任务，不知道来源单和目标单状态怎么闭环
3. 只看单套库存，不知道是谁驱动了这次节点变化

这份文档的目标就是把三层对象放进同一张逻辑图里。

---

## 2. 三层对象定义

### 2.1 第一层：订单层（业务单据）

对象：`Order`

关注的是：

1. 客户是谁
2. 预定日期是什么
3. 当前履约到哪一步
4. 是否已发货 / 已归还 / 已完成 / 已取消

订单层回答的问题：

1. 这笔业务单现在是什么状态？
2. 操作员下一步应该做什么？

---

### 2.2 第二层：转寄任务层（调度执行）

对象：

1. `TransferAllocation`
2. `Transfer`

关注的是：

1. 目标订单现在挂靠了哪个来源单
2. 是否已经生成待执行任务
3. 这次转寄任务有没有真正执行完

转寄层回答的问题：

1. 这张目标单是仓库发货还是转寄发货？
2. 若是转寄，来源单是谁？
3. 操作员是否需要去转寄中心完成闭环？

---

### 2.3 第三层：单套库存层（真实货物流转）

对象：

1. `InventoryUnit`
2. `UnitMovement`

关注的是：

1. 具体哪一套货现在在哪里
2. 这套货归属哪个订单
3. 最近一次物流单号是什么
4. 这套货经历了哪些节点

单套层回答的问题：

1. 某套货现在是在库还是在外？
2. 它从仓库出来后转寄了几次？
3. 它最终有没有回仓？

---

## 3. 三层对象之间的驱动关系

### 3.1 总关系图

```text
订单(Order)
  ├─ 决定业务状态（待处理/待发货/已发货/已归还/已完成）
  ├─ 产生挂靠需求（TransferAllocation）
  └─ 触发履约动作（发货 / 归还 / 完成）

转寄任务(Transfer / TransferAllocation)
  ├─ 决定来源单 -> 目标单的调度关系
  ├─ 完成时推动订单状态联动
  └─ 消耗挂靠锁，驱动单套库存切换归属

单套库存(InventoryUnit / UnitMovement)
  ├─ 承载真实货物流转
  ├─ 响应订单履约 / 转寄任务动作
  └─ 反向提供在外库存拓扑与异常看板
```

### 3.2 谁驱动谁

1. 订单层驱动“是否要发货、是否可取消、是否归还、是否完成”
2. 转寄层驱动“从谁转给谁、是否生成任务、是否完成转寄闭环”
3. 单套层是结果承载层，真实记录货怎么走

总结：

1. 订单是业务入口
2. 转寄是调度中枢
3. 单套库存是物理事实

## 4. 典型场景总图

### 4.1 场景A：仓库直发，最终回仓

```text
订单A
pending -> confirmed -> delivered -> returned -> completed

无转寄任务

单套U001
in_warehouse
 -> WAREHOUSE_OUT
 -> in_transit(current_order=A)
 -> RETURNED_WAREHOUSE
 -> in_warehouse
```

业务解释：

1. 订单A从仓库直接发出
2. 中间没有转寄
3. 客户用完后寄回仓库
4. 单套库存重新回到可用库存

---

### 4.2 场景B：仓库发给A，A再转寄给B

```text
订单A
pending -> confirmed -> delivered -> completed

订单B
pending -> confirmed/delivered -> delivered -> ...后续回仓

挂靠锁
A -> B : locked -> consumed

转寄任务T1
pending -> completed

单套U001
in_warehouse
 -> WAREHOUSE_OUT (到A)
 -> in_transit(current_order=A)
 -> TRANSFER_PENDING
 -> TRANSFER_SHIPPED (A -> B)
 -> TRANSFER_COMPLETED
 -> in_transit(current_order=B)
 -> 后续 RETURNED_WAREHOUSE
 -> in_warehouse
```

业务解释：

1. A 本来是仓库发货
2. 后续 B 进来后，A 被作为来源单挂靠给 B
3. 转寄任务完成后：
   - B 拿到货，成为新持有方
   - A 不再回仓，而是通过转寄任务完成闭环
   - 单套库存归属从 A 切换到 B

---

### 4.3 场景C：A -> B -> C 多次转寄

```text
订单A：仓库发货 -> 完成
订单B：接收转寄 -> 再次成为来源单 -> 完成
订单C：接收转寄 -> 最终回仓

单套U001
仓库 -> A -> B -> C -> 仓库
```

业务解释：

1. 同一套货会多次服务不同订单
2. 订单号会变，但单套编号不会变
3. 运营真正要看的，是这套货当前挂在哪个订单上

---

## 5. 关键闭环规则

### 5.1 订单层闭环

#### 仓库回仓单

1. `已发货 -> 已归还 -> 已完成`

#### 转寄来源单

1. 不在订单中心手动归还
2. 必须通过转寄任务完成闭环

这就是为什么：

1. 某些已发货订单在订单中心会显示“转寄中心操作”

---

### 5.2 转寄层闭环

1. 候选挂靠只是“计划”
2. 生成任务才是“待执行”
3. 完成任务才是真正的“调度闭环”

也就是说：

1. `TransferAllocation.locked` 不代表已经转走
2. `Transfer.completed` 才代表这次转寄真的执行完

---

### 5.3 单套层闭环

单套库存的闭环标准：

1. 最终出现 `RETURNED_WAREHOUSE`
2. 且 `InventoryUnit.status = in_warehouse`
3. 且 `current_order = null`

如果没有达到这个条件，就说明这套货仍在外面。

## 6. 三层对象的职责边界

### 6.1 哪些数据应该看订单层

适合在订单层看的：

1. 客户信息
2. 订单状态
3. 预定日期
4. 金额、押金、尾款
5. 当前是否挂靠转寄

不适合只靠订单层判断的：

1. 具体哪一套货在外
2. 一套货转寄了几次
3. 某 SKU 在外的真实链路拓扑

### 6.2 哪些数据应该看转寄层

适合在转寄层看的：

1. 推荐来源是谁
2. 当前挂靠是谁
3. 是否已经生成任务
4. 任务是否已完成/取消

不适合只靠转寄层判断的：

1. 单套库存目前具体处于什么节点
2. 单套是否已经回仓

### 6.3 哪些数据应该看单套层

适合在单套层看的：

1. 单套当前归属订单
2. 最近节点
3. 最近物流单号
4. 在外天数
5. 多次转寄路径

---

## 7. 三层一致性约束

系统后续开发必须满足以下一致性：

### 7.1 订单与转寄一致

1. 若订单显示“转寄挂靠”，则应存在有效 `TransferAllocation`
2. 若订单显示“已生成任务”，则应存在 `pending/completed` 的 `Transfer`

### 7.2 转寄与单套一致

1. 若转寄任务完成，则单套必须有：
   - `TRANSFER_SHIPPED`
   - `TRANSFER_COMPLETED`
2. 若挂靠锁 `consumed`，则必须能对应到任务完成

### 7.3 订单与单套一致

1. 若订单已回仓完成，则归属于该订单的单套不应仍停留在外
2. 若订单仍是来源单占用中，则单套当前归属可能还挂在该单或等待切换

## 8. 当前系统最重要的操作判断

### 8.1 在订单中心看到“标记归还”

说明：

1. 这张已发货订单应该回仓
2. 可以在订单中心完成闭环

### 8.2 在订单中心看到“转寄中心操作”

说明：

1. 这张单已经进入转寄链路
2. 不应在订单中心直接回仓
3. 需要通过转寄任务闭环

### 8.3 在在外库存看板看到单套仍在外

说明：

1. 不管订单层看起来是否结束，只要单套还没 `RETURNED_WAREHOUSE`
2. 这套货就仍然是“在途/在外”

## 9. 运营视角下最常见的问题映射

### 问题1：这张订单为什么不能点标记归还？

答案：

1. 因为它不是“回仓单”
2. 它在转寄链路中，必须到转寄中心处理

### 问题2：这套货到底现在在哪？

答案：

1. 去看 `InventoryUnit`
2. 看当前归属订单 + 最近节点 + 最近物流单号

### 问题3：来源单为什么已经完成了？

答案：

1. 因为它通过转寄任务完成了闭环
2. 并不代表货回仓，而是货被继续转给了下一个订单

### 问题4：为什么仓库可用库存没有回升？

答案：

1. 因为单套没有回仓
2. 它只是转寄给下一个客户了
