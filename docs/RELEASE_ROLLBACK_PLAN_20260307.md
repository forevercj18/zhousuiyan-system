# Phase A 发布与回滚预案（2026-03-07）

## 1. 发布范围
1. 角色看板数据层与前端动态渲染（`focus_cards / quick_actions / risk_entries`）。
2. 风险提示跳转联动（工作台 -> 订单中心/转寄中心/部件库存）。
3. 部件库存低库存筛选参数（`low=1`）。
4. 测试补强（API 与核心页面回归用例）。

## 2. 发布前检查清单
1. 执行：`python manage.py check`。
2. 执行：`python manage.py test -v 1` 全量通过。
3. 确认系统设置项存在默认值：`ship_lead_days / return_offset_days / buffer_days`。
4. 抽样验证 3 个角色登录工作台：
   - 管理角色显示经营视角卡片；
   - 仓库角色显示库存/转寄执行卡片；
   - 客服角色显示订单处理视角卡片。
5. 抽样验证风险入口跳转：
   - 待处理订单 -> `orders_list?status=pending`
   - 待执行转寄 -> `transfers_list?panel=tasks&status=pending`
   - 低库存部件 -> `parts_inventory_list?low=1`

## 3. 发布步骤
1. 代码发布到主分支（先打 tag 再发布）。
2. 应用数据库迁移（如本次发布包含迁移）。
3. 重启应用服务。
4. 按“发布前检查清单”做线上冒烟。

## 4. 监控与观察（发布后 30 分钟）
1. 重点监控 5xx、403、页面渲染错误。
2. 审计日志确认关键入口（订单、转寄、设置）持续写入。
3. 运营侧确认工作台卡片和快捷操作可正常点击。

## 5. 回滚触发条件
1. 核心链路失败：创建订单/确认发货/转寄任务生成任一失败率明显上升。
2. 高风险越权：权限控制出现放开或误拦截。
3. 数据展示错误：看板指标明显与订单中心统计不一致。

## 6. 回滚方案
1. 代码回滚到上一个稳定 tag（不改数据）。
2. 若本次有迁移且影响读写：
   - 优先采用“代码回滚 + 兼容读取”；
   - 非必要不做破坏性 `migrate rollback`。
3. 回滚后立刻执行：
   - `python manage.py check`
   - `python manage.py test apps.api.tests -v 1`
4. 验证关键页面可用：工作台、订单中心、转寄中心、部件库存。

## 7. 应急联系人与职责
1. 开发：负责快速定位和修复。
2. 测试：负责回归验证与复现记录。
3. 业务：负责线上可用性确认与放量建议。
